# ⚡ Flicker RealTime Messenger

## 📖 Обзор
**Flicker** — это полностековый веб-мессенджер, предназначенный для общения в реальном времени. Модуль решает задачи мгновенного обмена текстовыми сообщениями, поддержки P2P аудио/видеозвонков (WebRTC), использования анимированных эмодзи, а также интеграции с AI-ботами (OpenAI, GigaChat, DeepSeek, Gemini).

Целевая аудитория документа: **Frontend/Backend-разработчики** и **DevOps-инженеры**, занимающиеся поддержкой и развертыванием приложения.

---

## 🛠 Технологический стек
Проект не использует тяжелых фронтенд-фреймворков, полагаясь на модульный Vanilla JS, а бэкенд построен на асинхронном Python-стеке.

*   **Backend:** Python 3.11, FastAPI, Uvicorn, Jinja2 (для SSR базовых страниц).
*   **База данных:** MongoDB 6.0 (асинхронное взаимодействие, Motor).
*   **Frontend:** HTML5, CSS3 (CSS Variables для тем), Vanilla JavaScript (ES6 Modules).
*   **Real-Time & Media:** WebSockets (чат), WebRTC + TURN/STUN (звонки), FFmpeg (обработка медиа).
*   **Инфраструктура:** Docker, Docker Compose.
*   **Безопасность:** JWT (в куках и заголовках Authorization), CORS.

## 🏗 Архитектура
Проект разделен на две основные директории: `backend` и `frontend`. Взаимодействие между ними происходит через REST API, WebSockets и отдачу Jinja-шаблонов.

### Структура файлов (Ключевые компоненты)
*   **`backend/main.py`** — Точка входа FastAPI. Отвечает за инициализацию, смену рабочей директории (CWD) для шаблонизатора и подключение роутов.
*   **`backend/routes/`** — Бизнес-логика приложения:
    *   `websocket_manager.py` — Управление пулом активных WS-соединений.
    *   `calls.py` — Сигнальный сервер WebRTC (offer/answer/ice-candidates).
    *   `bots.py` — Интеграция с LLM-провайдерами.
*   **`frontend/chats/`** — Основной UI мессенджера:
    *   `js/modules/` — Изолированные скрипты: `calls.js` (менеджер WebRTC), `theme.js` (переключение тем), `avatar.js`.
    *   `js/main.js` — Инициализация UI, парсинг анимированных эмодзи и биндинг событий.
    *   `css/` — Модульные стили (`base.css`, `calls.css`, `design-chat.css`).

## 🚀 Основные сценарии использования (Task-based)

### 1. Как запустить проект (Локальная разработка)
Для запуска всех сервисов (FastAPI + MongoDB) используется Docker Compose. Окружение конфигурируется через `.env`.
```bash
# 1. Создайте .env файл в корне проекта на основе доступных переменных
# 2. Соберите и запустите контейнеры
docker-compose up --build -d
```
Приложение будет доступно по адресу: `http://localhost:8000`.

### 2. Как добавить нового AI-бота
1. Добавьте API-ключ нового провайдера в `.env` и `docker-compose.yml` (в секцию `environment`).
2. Создайте обработчик запросов в `backend/routes/bots.py`.
3. Зарегистрируйте бота в MongoDB (в коллекции пользователей/ботов, если боты выступают как отдельные сущности).
4. На фронтенде обновите список контактов, чтобы бот отображался с уникальным бейджем или аватаром (`images/avatars/`).

### 3. Как настроить WebRTC-звонки (Аудио/Видео)
Звонки работают через P2P. Если клиенты находятся за NAT, требуется TURN-сервер.
1. Получите учетные данные для TURN (например, от Metered.ca или поднимите свой `coturn`).
2. Пропишите переменные в `.env`:
   * `METERED_APP_NAME` и `METERED_API_KEY` (если используете облачный TURN).
   * Либо `TURN_URL`, `TURN_USERNAME`, `TURN_CREDENTIAL`.
3. Менеджер звонков на фронтенде (`frontend/chats/js/modules/calls.js`) автоматически запросит эти данные с бэкенда перед созданием `RTCPeerConnection`.

### Подключение к защищенному маршруту (Backend)
Аутентификация поддерживает чтение JWT как из `Authorization: Bearer`, так и из `Cookies` (`backend/dependencies.py`):

```python
from fastapi import Request, Depends
from backend.dependencies import get_current_user

@app.get("/api/secure-data")
async def get_data(current_user: dict = Depends(get_current_user)):
    # Если токен невалиден или отсутствует, Depends выбросит HTTP 401
    return {"user": current_user["email"]}
```

## ⚠️ Потенциальные сложности (Pitfalls)

1. **Jinja2 и Рабочая Директория (CWD):** 
   В `backend/main.py` принудительно изменяется рабочая директория на `frontend/` через `os.chdir(FRONTEND_DIR)`. Это сделано для того, чтобы Jinja2 корректно находил шаблоны. Если запускать скрипты из других директорий в обход `main.py`, пути к шаблонам могут сломаться.
2. **WebRTC в локальной сети:** 
   Без настроенных переменных `TURN_...` в `.env` звонки будут работать только в пределах одной локальной сети. Для звонков через интернет наличие TURN-сервера **обязательно**.
3. **Хранение медиа-файлов:** 
   Аватары и изображения загружаются в примонтированные volume Docker: `./images:/app/images`. Если вы пересоздадите контейнер без volume, пользовательские аватары будут утеряны. Убедитесь, что права на папку на хосте позволяют запись изнутри контейнера.
4. **Безопасность Cookies:** 
   Если переменная окружения `COOKIE_SECURE=true`, куки авторизации будут передаваться только по HTTPS. Для локальной разработки без SSL установите `COOKIE_SECURE=false`.
